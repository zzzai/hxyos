# ADR-002：HXYOS V1 总体架构

## 状态

Accepted

## 日期

2026-07-18

## 背景

HXYOS 服务荷小悦首店筹备、门店执行、员工成长和线上运营。它不是知识库前端、通用聊天机器人或 Agent 控制台，而是荷小悦内部的连锁经营执行与组织学习系统。

首店当前处于装修筹备阶段，内部沟通将统一迁移到飞书。系统必须同时满足：

- 普通员工低摩擦使用；
- 飞书沟通能沉淀为经营动作；
- 手机和电脑均可执行任务、上传证据和接受训练；
- AI 超时或失败不能阻塞业务事件创建；
- 核心知识、业务状态和经营指标不能由模型直接修改；
- HXY 数据与 htops、荷塘悦色保持严格隔离。

荷小悦的扩张路径已确定为：先由直营门店验证经营模型和组织复制能力，再根据证据开放加盟、联营、托管等混合模式。V1 必须避免把直营假设硬编码进核心事实，也不能为尚未验证的加盟业务提前建设完整功能。

## 产品原则

HXYOS 采用以下产品定位：

```text
AI 驱动
+ 规则治理
+ 人对高风险决策和经营结果负责
```

AI 不是可有可无的附加按钮。AI 默认参与信息理解、任务建议、风险检查、总结和复盘。但业务状态由状态机维护，权限由治理规则控制，正式知识和重大决策由授权人员核定。

这里的“AI 驱动”指组织运行方式，而不是让模型拥有业务控制权：

- 在功能层，AI 是可降级的辅助能力；模型不可用时，原始信息仍能留存，确定性流程仍可执行；
- 在产品层，AI 默认参与每一次适合机器理解和生成的工作，使沟通更快转化为任务、证据、复盘和组织方法；
- 在治理层，模型只产生 `AIProposal`，不能直接成为状态变化执行者、正式知识发布者或经营指标来源。

## 决策

### 1. 交互面

```text
飞书机器人 / 飞书群 / 响应式 PWA / 总部控制台
                         ↓
                    Channel Gateway
```

- 飞书是 V1 的组织沟通、通知、轻确认和任务触达主渠道。
- 荷小悦内部日常协作统一迁移到飞书；HXYOS 不复制另一套组织聊天系统。
- HXYOS 只处理指定群、`@HXYOS`、卡片动作、明确指令和授权工作流，不默认读取全部群聊或私聊。
- PWA 是现场采集、任务执行、证据提交、培训和历史查询的主工作面。
- 总部控制台负责组织配置、规则核定、审计、跨店分析和技术运行查看。
- 普通用户前端、总部控制台和技术运维入口使用独立路由、权限和前端构建入口。
- Hermes 可以作为可替换的理解或生成能力，不保存 HXY 权威业务状态。

### 2. 身份与权限

飞书身份映射到 HXYOS 身份，但飞书不是 HXYOS 权限权威源。

HXYOS 自己维护：

```text
Organization
LegalEntity
Store
StoreOperatingRelationship
GovernanceProfile
User
Assignment
Role
Permission
ChannelIdentityBinding
```

飞书事件进入系统前必须完成事件验签、租户校验、用户映射、消息去重和权限范围计算。

### 3. 应用架构

V1 使用 Python + FastAPI 模块化单体，不采用微服务拆分。

```text
identity       组织、门店、岗位、权限、渠道身份
channel        飞书/PWA入口、消息账本、附件接收
work           经营事件、工作流、任务、状态机
evidence       现场证据、验收、文件权限
training       员工训练与能力记录
service        顾客服务过程与反馈
content_ops    品宣、内容生产、风险预检、复盘
knowledge      正式知识、版本、引用和检索
catalog        资料、数据源、快照、血缘和数据分类
metrics        指标定义、经营事实、确定性计算和价值证据
ai_runtime     模型路由、多模态理解、AI建议和评测
jobs           Outbox、租约、重试、死信和人工重跑
```

新增领域模块直接按上述边界建设。现有大型 API 文件逐步迁移，不先进行与业务闭环无关的全面重构。

### 4. 核心运行方式

```text
渠道输入
→ InboundEnvelope 原始账本
→ 同一事务写入 OutboxMessage
→ Worker 异步调用 AI
→ AIProposal
→ 规则自动接受或触发例外确认
→ OperatingEvent / WorkflowInstance / Task
→ Evidence / StateTransition
→ MetricFact
```

AI 调用不得位于创建原始输入或确定性业务状态的数据库事务中。

### 5. 数据和文件

- PostgreSQL 16 保存组织、权限、经营事件、任务、状态、审计和指标事实。
- HXYOS 建立统一数据目录，但不把所有数据塞入同一张表或同一向量库。
- 文档、图片、音视频进入 `SourceAsset` 管线；POS、会员、支付、团购等结构化数据进入 `DataSource → DataConnector → DatasetSnapshot → BusinessFact` 管线；门店任务和问题进入 `OperatingEvent` 管线。
- `AssetBinding` 记录资料、数据集、知识、对话、经营事件和证据之间的来源血缘。
- `MetricDefinition` 版本化定义营收、客单、复购、退款率等业务口径，模型不能自行决定指标公式。
- pgvector 只索引受治理知识和明确允许检索的内容，不无差别向量化群聊、个人信息和经营事件。
- S3 接口对象存储保存图片、音频、视频、设计文件和现场证据。
- 文件对象记录哈希、来源、权限、生命周期、扫描状态和临时签名访问策略。
- 业务状态变化、证据和指标事实采用不可覆盖、可追溯的记录方式，不引入区块链或完整事件溯源框架。

### 6. 权威系统边界

HXYOS 是组织身份、正式知识、经营工作流、证据和治理状态的权威系统。

订单、会员、支付和团购平台初期继续由各业务系统保存权威原始记录。HXYOS 通过受控连接器或文件导入保存不可变快照、标准化经营事实和数据血缘，V1 不回写覆盖原系统交易。

未来是否建设自有 POS、会员或交易系统，必须由业务收益、数据控制需求和集成成本证明，不因 HXYOS 的“操作系统”定位自动纳入。

### 7. 异步任务

V1 继续使用 PostgreSQL 持久化队列，并把现有资料处理队列的成熟机制提炼为通用 Job Runtime：

- `FOR UPDATE SKIP LOCKED`；
- 幂等键；
- Worker 租约和过期回收；
- 最大重试次数和指数退避；
- 永久失败/死信状态；
- 超时和模型调用审计；
- 管理员人工重新运行。

当前不引入 Temporal。出现跨多日复杂补偿、大量定时器或多服务编排后，再根据运行证据决策。

### 8. 扩张与经营模式

HXYOS 采用直营验证后再开放加盟、联营、托管等混合模式。

- `Store` 保存稳定门店身份。
- `StoreOperatingRelationship` 版本化记录所有者、经营者、模式和有效期。
- `GovernanceProfile` 定义不同模式的决策权、审批权、数据共享、指标上报和审计要求。
- 所有模式共用品牌核心、服务安全底线和指标口径；权限、责任和数据范围可按合同差异化。
- 扩张准入由单位经济、顾客价值、人员复制、运营稳定、供应保障、数据完整和总部支持能力的证据决定，不按日期或门店数量自动触发。
- V1 只实现直营首店所需闭环，不提前建设加盟招商和加盟商管理功能。

### 9. 部署

```text
公网 Nginx
→ 飞书事件回调 / PWA / API
→ HXY 私有运行环境
→ FastAPI + Worker + PostgreSQL + S3接口对象存储
```

公网边缘不保存品牌知识和门店业务数据。所有服务、目录、数据库和备份均由 `/root/hxy` 对应的 HXY 环境独立拥有，不使用 `/root/htops`、htops API 或荷塘悦色业务数据库，不共享业务数据。

## 第一条纵向闭环

第一条完整业务闭环固定为：

```text
门店问题上报
→ 自动识别与分级
→ 责任分派
→ 整改证据
→ 店长验收
→ 高风险例外确认
→ 事件关闭
→ 指标事实
→ 重复问题形成标准候选
```

## 非目标

V1 不建设：

- 原生 iOS/Android App；
- 全量微服务架构；
- 由 Agent 自由控制业务状态；
- 无审核自动发布对外内容；
- 群聊自动升级为正式知识；
- 完整 CRM、ERP、BI 或自动化营销平台；
- 加盟招商、加盟商生命周期、区域代理和复杂结算系统；
- 自建 POS、支付或会员交易核心；
- 与 htops 共享业务数据。

## 后果

- 飞书成为组织协作入口，降低员工主动打开新系统的要求。
- PWA 保留深度工作能力，避免把复杂执行强塞进聊天卡片。
- AI 默认参与，但模型失败不会阻塞经营事件留存。
- 核心业务对象和证据链可以支持后续多店复制与价值证明。
- 统一数据目录和经营关系版本使直营证据可以迁移到未来混合经营模式，而不重写核心事实。
- 系统在首店阶段保持单体运维效率，同时保留按真实负载演进的边界。
