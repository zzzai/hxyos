# HXY Operating Result Delivery Design

## Product Principle

荷小悦经营大脑不是 AI 工具展示台，而是稳定交付经营结果的系统。客户不关心用了几个模型，只关心结果准不准、速度快不快、风险能不能降下来。

## Goal

把当前“经营问答”升级为“经营结果交付台”。每次回答都产出稳定结构的经营结果卡，降低输出方差，并把老员工经验、创始人判断、招商话术、门店 SOP 沉淀为组织资产。

## First-Phase Scope

第一阶段不做复杂智能体网络，也不扩展供应链、采购、关务等泛场景。先跑通荷小悦当前最高频的三个结果闭环：

1. 门店员工培训
2. 招商加盟问答
3. 产品体系和用户端表达

## Result Card Contract

API 每次回答都返回 `result_card`：

- `result_type`: 结果类型，例如品牌定位、产品体系、门店模型、招商话术、培训 SOP。
- `usable_answer`: 可直接使用版本，面向当前场景。
- `business_result`: 这次回答要交付的经营结果。
- `risk_boundary`: 风险边界，说明不能怎么用。
- `quality_gates`: 质量闸口，逐项说明是否通过。
- `review_owner`: 建议复核角色。
- `stability_level`: 交付稳定级别，分为 `stable`、`review_required`、`insufficient`。

## Quality Gates

第一阶段质量闸口固定为五项：

1. 命中正确业务域
2. 使用干净业务结论
3. 无内部路径、文件名、chunk、OCR 噪声
4. 适配当前场景
5. 不过度承诺

任何闸口失败时，答案不能伪装成确定结论，必须进入 `待复核` 或 `资料不足`。

## Frontend Shape

页面主标题保留“荷小悦经营大脑”，主工作区升级为“经营结果交付台”。

主聊天区展示：

- 经营结果
- 可直接使用版本
- 风险边界
- 质量闸口
- 可执行动作

来源、文件路径、chunk、OCR 和证据仍只出现在折叠的内部复核区。

## Non-Goals

- 不展示模型链路。
- 不做泛企业 AI 平台。
- 不把 POC 需求堆成定制外包。
- 不允许低质量召回被包装为业务结论。

## Success Criteria

- 高频问题输出结构稳定。
- 用户端宣传、招商话术、门店培训能拿到不同版本。
- 低置信度答案自动进入复核。
- 主答案不再出现内部路径、文件列表、无关资料。
- 前端能清楚显示结果卡、质量闸口和风险边界。
