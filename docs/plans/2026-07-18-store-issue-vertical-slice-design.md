# HXYOS Store Issue Vertical Slice V1 Design

## 目标

在首店装修筹备期跑通第一条真实经营闭环：把飞书群或PWA中的现场问题，转化为有责任人、有证据、有验收、有指标的经营事件。

该纵向闭环用于验证 HXYOS 的核心产品命题：

```text
组织沟通能否被AI转化为经营动作，
并在不依赖创始人反复追问的情况下形成可审计结果。
```

首条切片运行在直营首店，但业务事实不能硬编码“直营即全部权限”。每次创建事件、分派、查看证据和验收时，系统都读取门店当前 `StoreOperatingRelationship` 与 `GovernanceProfile`。V1 当前关系为 `direct_operated`，未来切换到加盟、联营或托管时不改写历史事件。

## 用户

```text
店长             现场上报、处理协调、证据提交、一般验收
创始人/总部负责人  只处理重大偏差、重大预算、品牌和安全事项
设计/施工方        V1不要求登录，由店长记录外部处理结果
系统管理员         身份映射、失败重跑、审计和策略配置
```

## 入口

### 飞书

支持以下最小交互：

- 在指定项目群中 `@HXYOS` 并发送文字、图片或文件；
- 对一条消息使用“上报问题”卡片动作；
- 接收任务分派、补充信息、到期提醒和验收卡片；
- 从卡片跳转PWA查看完整事件或上传更多证据。

飞书事件先进入 InboundEnvelope。机器人不直接写任务、正式知识或指标。

### PWA

PWA保留四个一级入口：

```text
对话
上传
今日
我的
```

用户可在主输入框中拍照、录音、上传文件或输入文字。选择“上报问题”后，系统立即保存原始输入并显示已接收状态，不等待模型返回。

“上传”是 HXYOS 自有的一等入口，不依赖飞书。用户可上传文件、图片、音频、视频和短文字，先形成组织级 `SourceAsset` 并进入“组织资料待处理区”。上传时可选择关联当前问题、任务、培训或仅保存待处理；未选择业务用途时不能由模型擅自创建经营事件或正式知识。

### 结构化数据入口

POS、会员、支付、团购平台和人工台账不走 `SourceAsset` 文档理解流程。它们通过 `DataSource`、`DataConnector` 或文件导入形成不可变 `DatasetSnapshot`，再标准化为 `BusinessFact`。第一条切片只建立可绑定边界，不接入完整交易系统。

## 主流程

```text
1. 店长在飞书或PWA提交问题和现场材料
2. 附件形成SourceAsset，原始消息形成InboundEnvelope，并通过AssetBinding建立来源关系
3. Channel Gateway完成验签、身份映射、去重和权限计算
4. 同一事务写入InboundEnvelope和OutboxMessage
5. Worker异步执行OCR/视觉/语音和问题提取
6. 系统保存AIProposal：标题、类型、严重度、责任人建议、缺失字段
7. 治理策略读取StoreOperatingRelationship和GovernanceProfile，决定自动接受、追问或升级
8. 创建OperatingEvent、WorkflowInstance和整改Task
9. 责任方处理，店长上传整改后Evidence
10. Task进入submitted，事件进入resolved
11. 店长或总部按风险策略验收
12. Task accepted，事件closed
13. 系统按MetricDefinition计算闭环时长、逾期、返工和验收次数
14. 重复问题只形成标准候选，不自动修改SOP
```

## 自动化策略

### 自动推进

满足以下条件时不要求人工确认分类：

- 来源用户和门店身份明确；
- 问题属于已发布类型；
- AI置信度达到策略阈值；
- 未命中安全、品牌、预算、医疗、投诉等高风险规则；
- 责任人可由门店和问题类型规则确定。

### 只追问缺失信息

系统仅针对阻塞流程的字段追问，例如：

- 无法判断具体位置；
- 没有可验收的完成标准；
- 责任方不明确；
- 缺少必要现场证据。

### 强制升级

以下情况至少升级到总部负责人：

- 人员安全或施工安全；
- 重大预算和工期变更；
- 与已核定VI/SI或品牌核心明显冲突；
- 证照与合规风险；
- 同类问题重复返工；
- 店长与责任方对验收结论存在争议。

## 飞书卡片

### 问题已识别

显示：

```text
问题摘要
门店/位置
严重度
责任人
截止时间
需要补充的信息
```

动作：`确认并推进`、`修改`、`不是问题`、`查看详情`。

低风险高置信度事件不显示确认卡，只通知已创建和责任人。

### 待验收

显示整改前后证据和完成说明。

动作：`验收通过`、`退回整改`、`升级总部`。

### 每日摘要

只显示：

```text
今日新增
今日关闭
逾期事项
重复返工
需要创始人决策
```

## 权限

- 店长只能查看和处理其门店范围内的事件。
- 技师只能查看分派给本人或对其岗位公开的事项。
- 创始人/总部负责人按组织权限查看跨店事件。
- 经营关系不是前端可信字段，服务端按事件发生时的 `GovernanceProfile` 计算数据范围和验收人。
- 技术管理员可查看运行状态，但默认不能读取超出运维所需的业务正文。
- 飞书群中的卡片内容按群成员权限最小化展示，敏感详情跳转PWA后再次鉴权。

## AI边界

AI可以：

- 识别图片、语音、文本和文件；
- 提取问题、位置、影响和完成标准；
- 建议严重度、责任人和截止时间；
- 汇总进度并发现重复问题；
- 生成整改说明草稿。

AI不可以：

- 伪造现场证据；
- 作为正式状态变化的执行者；
- 自动关闭高风险事件；
- 修改正式知识或VI/SI标准；
- 自动生成MetricFact的数值事实。

## 失败处理

- 飞书重复回调由渠道幂等键消除。
- AI超时不影响InboundEnvelope留存，前端显示“已接收，正在理解”。
- Worker按指数退避重试，超过上限进入死信并通知管理员。
- 附件解析失败时保留原文件，允许人工重新运行或补充说明。
- 身份无法映射时不创建业务事件，进入受限的身份待处理队列。
- 卡片回调必须验证当前状态，重复点击返回现有结果。

## 验收标准

使用真实首店装修事项完成以下验收：

1. 飞书和PWA均可在30秒内完成一次问题上报。
2. 模型超时或不可用时，原始输入仍可靠保存且不重复。
3. 至少10个真实问题完成从上报到验收关闭的完整闭环。
4. 每个关闭事件都有责任记录、状态轨迹和至少一个有效结果证据。
5. 低风险高置信度事项无需人工确认分类。
6. 高风险事项不能由AI或普通员工单独关闭。
7. 闭环时长、逾期时长、返工次数由状态记录确定性计算。
8. 创始人视图只出现待决策、重大风险和长期阻塞事项。
9. 飞书重复事件、重复卡片操作和PWA重复提交均保持幂等。
10. 群聊和经营事件不会自动进入正式知识库。
11. HXYOS“上传”入口可以独立于飞书建立SourceAsset，并按权限预览和绑定业务用途。
12. 所有关闭事件保留事件发生时的经营关系、治理策略和指标口径版本。

## 非目标

- 全量读取所有飞书私聊和群聊；
- 自动管理外部设计或施工公司的飞书账号；
- 完整装修项目管理系统；
- 加盟管理功能、招商流程和混合模式结算；
- 自动批准预算和重大变更；
- 自动更新SOP；
- 对外内容自动发布；
- 与htops共享任何业务数据。
