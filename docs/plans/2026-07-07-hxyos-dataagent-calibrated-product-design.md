# HXYOS DataAgent-Calibrated Product Design

## Purpose

This document updates the HXYOS product direction after reviewing the Volcengine DataAgent private deployment documentation for Data Agent, Enterprise Knowledge Engine, Marketing Interaction Assistant, Skill, MCP, memory, retrieval applications, automation tasks, Feishu integration, and permission management.

The goal is not to copy Volcengine's product. The goal is to use a mature enterprise AI product as a pressure test for HXYOS:

```text
If HXYOS is an enterprise-grade long-memory operating brain,
then knowledge, retrieval, skill, intent, memory, permission, automation, and audit
must be first-class product objects.
```

## Key Reading

The reviewed Volcengine documentation validates several product patterns:

- Enterprise AI is not a chat window. It is a connected system of data, knowledge, tools, agents, skills, memory, security, and operations.
- Knowledge management is not file upload. It includes import, parsing, segmentation, tagging, status, effective period, versioning, retrieval testing, and monitoring.
- Retrieval should be publishable as an application with tunable parameters, API access, and debugging, not hidden inside a generic `/ask` endpoint.
- Agent orchestration needs explicit intent planning, agent registration, tool registration, skill lifecycle, memory policy, and safety gates.
- Memory should be generated by strategies and used under read/write rules. Chat history must not mutate official knowledge.
- Feishu and group collaboration must respect permissions. Group chat cannot become an uncontrolled private-data query surface.
- Automation tasks need schedules, execution records, timeout, result channels, and human-visible state.

## Product Thesis Update

The previous HXYOS PRD correctly rejected "AI knowledge base + chat" as the product center. The DataAgent review sharpens the target:

```text
HXYOS = 荷小悦 Knowledge Engine
      + Operating Agent Router
      + Skill Registry
      + Governed Memory Layer
      + Retrieval App Layer
      + Loop Automation Console
      + Review And Audit System
```

The frontstage can remain minimal:

```text
一个问答框
+ 场景化工作流入口
+ 当前经营议题
+ 可交付结果
```

The backstage must be serious:

```text
知识治理
检索应用
技能版本
意图规划
记忆策略
权限控制
自动化任务
评测与审计
```

## Approaches Considered

### Approach A: Keep The Current Knowledge Workbench

Continue improving the current compiler, review queue, benchmark, and answer-card flow.

Pros:
- Lowest engineering cost.
- Uses the current artifacts and tests.
- Good for short-term cleanup.

Cons:
- Still feels like a technical admin page.
- Does not introduce retrieval applications, intent planning, skills, or automation as product objects.
- The frontstage remains weak and the backend artifacts leak into the UI too easily.

### Approach B: Replace HXYOS With A Third-Party RAG/Agent Platform

Adopt a platform like RAGFlow, Dify, OpenRAG, or a cloud DataAgent-style product as the main system.

Pros:
- Faster demo surface.
- Mature parsing or workflow pieces may be reused.
- Less initial platform building.

Cons:
- Weakens HXY ownership of core knowledge and process memory.
- Harder to enforce HXY-specific approval, compliance, and local-data boundaries.
- Does not solve the product judgment problem; it only changes the tool layer.

### Approach C: Keep HXYOS, Upgrade Product Objects

Keep the current HXY-owned foundation, but promote mature enterprise AI primitives into explicit HXYOS objects.

Pros:
- Preserves HXY data boundary and governance.
- Fits the existing compiler, benchmark, loop, memory, and review work.
- Gives the frontend a better content model: business workflows and review topics, not raw machine claims.
- Lets HXYOS integrate third-party engines later through adapters instead of becoming dependent on them.

Cons:
- Requires a PRD and model refactor before UI work feels right.
- More product discipline is needed: every object needs lifecycle, owner, and permission rules.

Recommendation: **Approach C**.

## Final Design

### 1. Knowledge Engine

The Knowledge Engine replaces the vague idea of a "knowledge base".

It contains:

- `knowledge_library`: named knowledge collections by content type and domain.
- `knowledge_asset`: original or imported files, links, tables, transcripts, and external references.
- `knowledge_segment`: parsed chunks, sections, tables, images, and extracted units.
- `knowledge_tag`: business labels for domain, use case, risk, role, source, and validity.
- `knowledge_version`: version history for each managed asset.
- `knowledge_run`: import, parsing, tagging, segmentation, and compiler run records.

HXY-specific content types:

| Type | Use |
|---|---|
| `general_content` | books, articles, SOP, strategy, training, reports |
| `approved_qa` | approved standard answers |
| `term_library` | HXY terms, aliases, forbidden phrases, industry expressions |
| `query_cache` | high-frequency stable questions, with approved response patterns |
| `operating_data` | future HXY-owned store, service, training, and customer-feedback facts |

Rules:

- Raw and reference material cannot answer as authority.
- Approved QA and approved action assets can answer as authority.
- External references can shape questions and hypotheses, but cannot become HXY conclusions without review.

### 2. Retrieval Application Layer

HXYOS should not expose only a generic answer pipeline. It should support publishable retrieval applications.

Examples:

- `employee_standard_answer_search`
- `brand_language_risk_check`
- `opening_store_sop_search`
- `founder_decision_evidence_search`
- `compliance_reference_search`

Each retrieval app has:

- name and description;
- referenced knowledge libraries;
- allowed tags and excluded tags;
- priority weights;
- lexical/vector mix;
- similarity threshold;
- rerank strategy;
- initial candidate count;
- final segment count;
- smart query rewrite on/off;
- debug questions;
- status: draft, published, disabled;
- API token or internal service binding.

Why this matters:

```text
Different HXY jobs need different retrieval behavior.
员工话术要保守、短、只查 approved。
创始人战略判断要能查 reference、candidate、历史复盘，但必须标注不确定性。
合规检查要优先召回禁用表达和风险规则。
```

### 3. Intent Planning Engine

The first layer after user input should be intent planning, not direct answer generation.

HXYOS should recognize:

- ask an approved answer;
- upload/reference material ingestion;
- ask for decision support;
- request content draft;
- request compliance review;
- request employee training;
- request brand wording review;
- request loop execution;
- correction or dispute;
- memory promotion candidate.

Each intent must define:

- positive scope;
- excluded scope;
- example questions;
- required role;
- allowed retrieval apps;
- allowed skills;
- risk gates;
- fallback behavior.

For multi-intent input, HXYOS should either clarify or plan a sequence. It should not silently blend conflicting tasks.

### 4. Skill Registry

Skill is the reusable ability unit above tools and below agents.

First HXY skills:

- `hxy-compliance-language-check`
- `hxy-brand-expression-review`
- `hxy-employee-answer-coach`
- `hxy-ingest-material-compiler`
- `hxy-opening-store-checklist`
- `hxy-benchmark-correction-pack`
- `hxy-decision-evidence-pack`
- `hxy-review-topic-generator`

Each skill has:

- name;
- business description;
- owner;
- input schema;
- output schema;
- allowed tools;
- required retrieval apps;
- prompt or rule contract;
- examples;
- test cases;
- status: draft, published, disabled;
- version;
- changelog;
- rollback path.

Rules:

- Published skills may be used by agents.
- Draft skills can be tested but not used in official workflows.
- Skills with external actions require stronger review.
- Skill output is not official knowledge unless it goes through publishing.

### 5. Governed Memory Layer

HXYOS memory should stay separate from official knowledge.

Memory types:

- `working_memory`: current task state, plan, tool results, stop condition.
- `short_term_memory`: current thread summary and recent constraints.
- `event_memory`: what happened in a workspace, review, meeting, loop, or store event.
- `profile_memory`: stable role preference, user/team preference, entity profile.
- `procedure_memory`: how HXYOS should do a task, stored as versioned skill or workflow rule.

Memory policy fields:

- read allowed;
- write allowed;
- global or domain memory;
- retention/TTL;
- extraction strategy;
- compression strategy;
- confidence;
- promotion status;
- conflict status.

Rules:

- AI answers may use process memory only as context hints.
- Process memory cannot be a formal citation.
- Chat cannot modify Core Knowledge Vault.
- Memory promotion creates a proposal, not approval.

### 6. Automation Task Layer

Loop Engine should become a visible product layer, not just scripts.

Automation task types:

- script task;
- ingest loop task;
- benchmark loop task;
- retrieval app evaluation task;
- review queue triage task;
- workspace digest task;
- Feishu/Hermes notification task.

Each task has:

- schedule: manual, interval, cron, one-time;
- task input;
- timeout;
- enabled/disabled;
- last run state;
- execution records;
- output artifacts;
- webhook/Feishu notification target;
- stop condition.

Rules:

- Tasks cannot approve formal knowledge.
- Tasks cannot run arbitrary shell commands from UI.
- Tasks must show execution records and stop reasons.

### 7. Feishu/Hermes Interaction

Feishu and Hermes should be channels, not authority layers.

Recommended behavior:

- Web app remains the primary governed workspace.
- Hermes/Feishu can submit questions, files, corrections, and loop triggers.
- Single chat can query according to user permission.
- Group chat defaults to public-safe functions.
- Sensitive knowledge requires permission check and may return a redacted answer.
- Group chat should be excellent for organization learning, task notifications, and shared issue tracking, not uncontrolled private knowledge lookup.

### 8. Frontend Product Shape

The current frontend issue is content, not only styling. The page should not display raw backend artifacts as primary content.

Frontstage:

```text
顶部：一个问答框
下方：场景工作流
中部：当前经营议题 / 下一步动作 / 可用答案
侧边：引用、风险、状态
```

Backstage:

```text
知识引擎
检索应用
意图配置
Skill
记忆策略
Loop 自动化
复核发布
Benchmark
审计
```

Raw candidate claims, chunk fields, file paths, and cluster counters must stay internal.

## Data Model Additions

New first-class objects:

```text
knowledge_library
knowledge_asset
knowledge_segment
knowledge_tag
retrieval_app
intent_definition
skill_definition
memory_policy
automation_task
execution_record
agent_registration
channel_binding
```

These objects can start as JSON-backed contracts and migrate to PostgreSQL later.

## Impact On Current Roadmap

Immediate correction:

- Stop treating review queue as the main UI content.
- Convert raw claims into review topics and business decision tasks.
- Add product docs for retrieval apps, skill registry, intent planning, and automation tasks.

Next implementation:

1. Add PRD v1.1 sections for the new objects.
2. Add API contracts for retrieval apps, skill definitions, and intent definitions.
3. Add admin HTML sections that show business objects, not raw compiler internals.
4. Add tests to keep raw artifacts out of user-facing pages.
5. Add benchmark cases that evaluate retrieval app behavior and intent routing.

## Non-Goals

This design does not require:

- replacing HXYOS with Volcengine;
- integrating Volcengine APIs now;
- building all enterprise UI pages immediately;
- storing external docs as HXY authority;
- bypassing current HXY security and data boundaries;
- exposing review/approval work in the public frontdesk.

## Acceptance Criteria

The product direction is corrected when:

- PRD names Knowledge Engine, Retrieval Apps, Skill Registry, Intent Planning, Memory Policy, Automation Tasks, and Review/Audit as first-class objects.
- Frontend-facing content uses business topics and workflows, not raw claim fields.
- Approved knowledge remains protected from chat, agent, loop, and memory writes.
- Retrieval behavior is configurable per business use case.
- Skills are versioned, testable, and publishable.
- Feishu/Hermes are channels with permission checks, not alternate unmanaged brains.
