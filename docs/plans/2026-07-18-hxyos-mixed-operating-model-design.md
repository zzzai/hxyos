# HXYOS 直营验证后混合经营模式设计

## 状态

Approved

## 决策

荷小悦采用“直营验证后再开放混合扩张”的路径：

```text
直营首店建立经营基线
→ 多家直营店验证可复制性
→ 通过扩张证据门槛
→ 分阶段开放加盟、联营、托管等经营模式
→ 持续按经营模式比较质量、风险和收益
```

开放下一种模式不以日期或门店数量自动开放。必须由经营证据证明单店模型、品牌一致性、服务质量、人才训练、供应链和总部支持能力已经达到已发布门槛。

## 为什么不能只给门店增加一个模式字段

一个 `operating_mode` 字段不能表达真实经营关系。门店可能由不同主体出资、经营、持有品牌授权、承担雇佣责任、使用会员数据并接受总部审计；这些关系还会随合同续签或经营权变更而变化。

因此必须把稳定的门店身份与版本化经营关系分离：

```text
Store
→ StoreOperatingRelationship
→ GovernanceProfile
```

- `Store`：稳定的经营地点和业务主体标识，不因合作模式变化而重建。
- `StoreOperatingRelationship`：记录所有者、实际经营者、模式代码、生效区间和合同引用。
- `GovernanceProfile`：记录品牌标准、决策权、审批权、数据共享、指标上报和审计要求。

V1 不把未来全部模式固化为数据库枚举。模式代码由版本化目录管理，首批预留：

```text
direct_operated          直营
franchise_operated       加盟商经营
joint_venture_operated   联营/合资经营
managed_operated         委托或托管经营
```

新增模式必须先定义责任、权限、数据和指标契约，不能只增加一个展示标签。

## 直营验证门槛

直营阶段的目的不是证明“可以开店”，而是建立可复制经营证据。至少覆盖：

```text
单位经济：收入、毛利、现金流、回本假设和敏感性
顾客价值：复购、投诉、退款、满意度和服务安全
品牌一致性：正式口径、服务体验和内容风险
人员复制：招聘、培训、授权上岗和能力保持
运营稳定：任务执行、异常闭环、返工和店长管理半径
供应保障：产品、耗材、质量、交付和异常替代
数据质量：关键事件、订单事实和指标字段完整可追溯
总部能力：支持响应、巡检、训练、风控和问题升级能力
```

每项门槛必须由 `MetricDefinition` 和可追溯事实计算。模型可以解释结果，不能制造通过结论。

## 数据与权威边界

HXYOS 的 System of Record 边界如下：

```text
HXYOS 权威：
组织身份、岗位权限、正式知识、工作流、任务、证据、状态、治理策略、指标定义

外部系统初期权威：
订单、会员、支付、退款、团购核销、平台流量和原始财务流水

HXYOS 保存：
不可变 DatasetSnapshot、标准化 BusinessFact、来源血缘、指标结果和经营解释
```

HXYOS 初期不替代 POS、支付、会员或团购平台，也不回写修改这些系统的原始交易。是否建设自有交易系统，必须由未来业务收益、数据控制和集成成本证明，而不是因为“操作系统应该什么都做”。

## 不同经营模式的数据原则

- 所有模式使用同一品牌核心、服务底线和 `MetricDefinition`。
- 直营店默认向总部提供完整经营事实和必要原始证据。
- 加盟、联营和托管门店依据合同应用不同的 `GovernanceProfile`，但不能降低品牌安全和最低审计要求。
- 顾客个人信息遵循最小必要访问；总部分析优先使用去标识化事实。
- 跨模式比较必须同时展示经营模式、数据覆盖率和口径版本，避免错误归因。
- 模式变更生成新的 `StoreOperatingRelationship` 版本，历史事实保持原经营关系快照。

## 产品表现

普通员工和店长不需要理解复杂扩张模型。前端仍以角色任务为中心：

```text
对话
上传
今日
我的
```

经营模式只在影响责任、审批、数据访问或结算时出现。总部控制台负责：

- 查看每家店当前经营关系和治理版本；
- 按模式比较经营质量、风险和数据完整度；
- 查看扩张门槛证据；
- 发布或撤销某种经营模式的准入；
- 处理合同、品牌和重大经营例外。

## 分阶段范围

### V1：直营证据闭环

直营首店只实现：统一数据目录、角色权限、任务与异常闭环、训练、资料上传、证据、正式知识和确定性指标。系统可以保存 `direct_operated` 经营关系，但不开发加盟管理功能。

### V2：多家直营复制验证

验证同一标准能否迁移，识别跨店差异，建立总部支持能力和扩张门槛报告。

### V3：受控混合模式试点

选择少量加盟、联营或托管门店，启用差异化 `GovernanceProfile`、合同数据边界和跨模式评测。

### V4：规模化治理

依据试点证据扩展模式，增加加盟生命周期、区域治理、供应链、结算和风险能力。

## 旧资料治理

历史资料或自动抽取结果中出现的“立即开放单店加盟”“首年快速开店”等表述，只能保留为历史参考或待验证假设，不能作为正式战略、招商承诺或系统默认配置。

本设计是当前扩张路径的权威决策。后续变更必须生成新的决策版本，不得用聊天记录或单份上传资料覆盖。

## 对首条纵向闭环的影响

门店问题闭环仍然是第一条业务切片，但在写入经营事件前必须具备：

1. 组织级 `SourceAsset` 与权限范围；
2. 门店当前 `StoreOperatingRelationship`；
3. 当前 `GovernanceProfile`；
4. 可追溯的资料、数据集、证据和经营事件绑定；
5. 版本化 `MetricDefinition`。

这样首店数据未来可与加盟、联营或托管门店比较，而无需重写核心业务事实。
