# HXY Understanding Engine Design

## Product Positioning

荷小悦 Understanding Engine 不是一个摘要器，也不是把资料切片后丢给大模型的搜索增强层。它是经营大脑的认知中枢：把任何输入先判断意图，再按业务认知模型理解，最后输出可执行、可追踪、可复核、会进化的经营结果。

它不和 GPT、Claude、DeepSeek 或 Qwen 比通用理解力。通用模型负责聪明，Understanding Engine 负责让模型懂荷小悦、遵守荷小悦边界、沉淀荷小悦记忆，并降低同类问题的输出方差。

## Core Model: Depth x Application

认知内核采用二维模型，而不是单线性的 0-6 层。

深度维度负责理解，必须递进：

- `D1_perception`: 感知关键词、实体、数字、文件类型、图片类型和输入形态。
- `D2_classification`: 判断业务域、对象、角色、场景和输入意图。
- `D3_decomposition`: 拆出事实、对象、属性、流程、限制条件、数据和关系。
- `D4_causal_inference`: 判断因果、业务影响、转化阻力、执行约束和可能后果。
- `D5_judgment`: 识别主要矛盾、关键杠杆、冲突元素和优先级。

应用维度负责交付，理解到 D5 后可以并发展开：

- `A1_role_output`: 面向总部、加盟商、店长、员工、顾客生成不同表达。
- `A2_risk_boundary`: 标注禁用表达、过度承诺、价格政策、医疗合规和不确定性。
- `A3_action_plan`: 给出下一步动作、责任人、时间、资源、能力和权限校验。
- `A4_conflict_correction`: 检测新旧资料冲突、理论与实操冲突、角色利益冲突，并生成复核任务。
- `A5_memory_evolution`: 沉淀答案卡、版本历史、热度、盲区、低质知识和纠偏信号。

## Intent Recognition Layer

所有输入先进入意图识别层。系统必须判断：

- 输入形态：文本、文件、图片、语音、链接、截图、表格、混合输入。
- 用户动作：提问、上传资料、执行指令、纠偏反馈、要求沉淀、要求生成物料。
- 业务需求：快速答案、深度分析、决策支持、员工培训、招商话术、用户表达。
- 紧急程度：日常咨询、运营问题、风险问题、战略讨论。

意图识别决定后续路径：

- 快速应答：优先查已批准答案卡。
- 深度理解：进入 D1-D5 认知管线。
- 执行指令：调用对应 Operating Skill。
- 入库记忆：进入资料识别、分类、评分、结构化和复核流程。
- 纠偏复核：生成冲突记录、复核任务和答案卡修订建议。

## Conflict Element

冲突是构成元素的一部分，不是事后备注。D3 拆解时必须显性抽取冲突元素，D5 判断时必须决定主次。

冲突类型包括：

- 新旧知识冲突：价格、政策、话术、版本、适用场景不一致。
- 总部门店冲突：总部标准和门店真实资源、能力、客诉、执行成本冲突。
- 理论实操冲突：战略逻辑成立，但员工不会讲、顾客不买单、门店无权限。
- 角色利益冲突：总部、加盟商、店长、员工、顾客的目标不一致。
- 合规表达冲突：业务想表达功效，但不能形成治疗、保证、绝对化承诺。

## Priority Matrix

主要矛盾不能靠语言感觉判断。第一版采用可解释的优先级矩阵：

```text
priority = impact * urgency * controllability * strategic_relevance
```

评分含义：

- `impact`: 对营收、转化、复购、品牌口径、培训效率、风险的影响面。
- `urgency`: 是否影响当前上线、门店执行、招商沟通或风险处理。
- `controllability`: 荷小悦团队是否能用资料、话术、SOP、系统或培训快速改变。
- `strategic_relevance`: 是否贴近清泡调补养、社区小店、加盟复制、经营大脑等核心战略。

系统输出主要矛盾时，必须同时给出排序理由和可控杠杆，避免只生成漂亮判断。

## Executability Gate

角色化方案不是完成点，必须经过可执行性校验。

校验维度：

- 资源：是否需要物料、系统、预算、数据或人员。
- 能力：员工、店长、招商、运营是否能理解并执行。
- 权限：门店是否有权改价、改话术、改流程、发优惠。
- 风险：是否涉及疗效承诺、价格政策、加盟收益、隐私合规。
- 验收：是否能被数据或人工验收，比如转化率、复购率、员工背诵通过率。

输出不能只写“建议培训员工”，必须转成“谁在什么时候用什么材料完成什么动作，如何验收”。

## Confidence And Evidence

主聊天区默认只展示可用答案，但系统内部必须保存置信度和证据链。

每个结果都要区分：

- 确定事实：来自已批准答案卡或高质量资料。
- 稳定判断：多资料一致，且无关键冲突。
- 推测建议：基于方法论或经验，需要验证。
- 待复核内容：价格、政策、收益、合规、医疗功效、门店执行条件。

置信度不是 UI 装饰，而是影响记忆权重、复核优先级和是否允许沉淀为权威答案。

## Versioned Organizational Memory

组织记忆必须像代码一样有版本和变更原因。

核心对象：

- 原始输入：用户上传或聊天提交的原始文件、图片、语音、链接和文本。
- 理解记录：D1-D5 的结构化结果、冲突元素、优先级评分和执行校验。
- 答案卡：面向高频问题的权威答案，含状态、版本、适用角色和适用场景。
- 纠偏记录：谁指出问题、哪里错、如何修、是否影响历史答案。
- 使用信号：调用次数、有用率、不准确率、复核次数、盲区问题。
- 版本历史：何时更新、为什么更新、旧版本何时仍然适用。

## Knowledge Evolution Layer

系统必须能自我诊断，而不是只等待用户反馈。

第一版演进信号：

- 热点知识：被频繁提问或调用的答案卡。
- 不稳定知识：反复被修改、被纠偏或资料冲突的知识点。
- 知识盲区：被多次问到但没有权威答案的问题。
- 低质资料：OCR 噪声、表格残片、元数据文件清单、缺少上下文的图片。
- 高价值待沉淀：高频且已经稳定回答的问题，应转成权威答案卡。

演进输出：

- 自动生成补资料请求。
- 自动生成复核任务。
- 自动建议答案卡升级、降权或废弃。
- 自动更新角色化话术模板。

## Multimodal Intake

上传不应只存在于侧边栏按钮。聊天框本身就是多模态入口。

输入支持：

- 文本：问题、资料片段、纠偏意见、指令。
- 文件：PDF、Word、PPT、Excel、Markdown、TXT。
- 图片：菜单图、竞品图、流程图、系统截图、报表截图、现场照片。
- 语音：培训录音、会议纪要、企微语音。
- 链接：网页、公众号、竞品页面。

图片理解采用组合策略：

- OCR 提取硬文字、价格、表格和标签。
- VLM 理解画面结构、业务含义、关系和场景。
- Layout/Table Parser 保留版式、表格和区域结构。
- 企业术语表把识别结果对齐到荷小悦业务语言。

## Product Output

用户看到的是极简经营结果，不是技术链路。

默认输出：

- 核心结论
- 主要矛盾
- 建议怎么做
- 风险边界
- 下一步动作

详情 Inspector 展示：

- 理解深度结果
- 应用维度结果
- 冲突元素
- 优先级矩阵
- 置信度
- 证据链
- 版本记录
- 复核与沉淀动作

## Non-Goals

- 不从头预训练模型。
- 不把所有能力塞进一个 Prompt。
- 不让主聊天区暴露 chunk、OCR、文件路径或模型链路。
- 不用 0-6 层作为系统逻辑。
- 不把低置信度生成内容伪装成权威结论。

## Success Criteria

- 同一输入能稳定识别是提问、资料、指令、纠偏还是沉淀请求。
- 任意资料能形成 D1-D5 理解记录。
- 高频业务问题能同时输出角色化版本、风险边界、行动建议、冲突检测和记忆沉淀建议。
- 主要矛盾有可解释优先级，不只是模型主观判断。
- 低置信度、冲突、过期和高风险内容自动进入复核。
- 使用反馈能反哺答案卡、资料权重和知识盲区清单。
